# Bounded autonomy leads the positioning

## Rule

Lead emisar positioning with the operator outcome: an MCP-capable agent can keep
doing useful infrastructure work without a human supervising every step or granting
open-ended production authority.

Earn that promise in this order when the surface has room:

1. **Mechanism:** declared actions, policy, pack trust, and runner-side validation
   bound what the agent can do.
2. **First value:** prebuilt packs and host-aware suggestions make useful actions
   available before a team has modeled every procedure itself.
3. **Extensibility:** the MCP tool surface stays fixed while packs and custom actions
   add infrastructure capabilities.
4. **Supporting controls:** approvals when policy requires them, audit, the host
   journal, and SIEM export make the system governable and reviewable.

Do not make approvals the centerpiece. An approval queue is a familiar control, not
the distinctive product outcome. The value is useful agent work continuing inside a
bounded action system.

## Why

Feature-first copy understates the actual job. Operators are not shopping for one
more approval inbox; they want to stop babysitting routine agent work without
quietly turning a model into a production shell user. The action catalog and fixed
MCP surface also explain why emisar delivers value sooner and scales beyond one
integration.

This hierarchy keeps the promise differentiated while preserving security
credibility. It connects the outcome to mechanisms a technical buyer can inspect.

## Good

- Start with the work the operator can leave running, then show why the authority is
  bounded.
- Show a prebuilt or host-suggested action as part of onboarding and time to value.
- Describe approvals as a conditional policy outcome: allowed, denied, or waiting
  for a person.
- Explain the extensibility claim concretely: install the MCP integration once, then
  add capabilities as packs and actions.
- Put the exact limit beside the security claim when it changes the decision.

## Bad

- Headline the product as an approval workflow, approval inbox, or audit dashboard.
- Suggest every action waits for approval; useful low-risk work should continue
  according to policy.
- Claim emisar runs the autonomous agent loop. It provides the governed action
  surface the agent calls.
- Promise that production cannot be damaged. emisar is not a sandbox, and permitted
  destructive actions still have real effects.
- Call emisar literally the last MCP a team will ever install. State the concrete
  design goal: one stable infrastructure MCP surface that grows through actions.

## Enforcement

Use `content-director` for positioning and `security-engineer` for trust claims.
Apply the hierarchy test in the content skill, then sweep adjacent marketing,
product, SEO, and creative-direction guidance for approval-first framing. This rule
depends on editorial judgment and verified product behavior, so enforce it in review
rather than with banned-word checks.
